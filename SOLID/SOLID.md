# SOLID 원칙
 - 유지보수하기 좋은 깔끔한 코드를 짜기 위한 5가지 규칙
 - IoC/DI, AOP, PSA가 왜 필요한지를 설명해 주는 근본 원리

# S(Single Responsibility Principle)
 - 단일 책임 원칙으로 하나의 책임(기능)만을 가져야한다.
 - ex) 요리사는 요리만 해야하지 배달까지 하면 안된다.

# O(Open/Closed Principle)
 - 개방 폐쇄 원칙으로 확장에는 열려있고 수정에는 닫혀 있어야 한다.
 - 새로운 기능을 추가할 때 기존 코드를 건드리지 않고 추가할 수 있어야한다.

# L(Liskov Substitution Principle)
 - 리스코프 치환 원칙으로 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.
 - 다형성을 사용할 때 예상치 못한 버그가 없어야 한다는 뜻

# I(Interface Segregation Principle)
 - 인터페이스 분리 원칙으로 자신이 사용하지 않는 기능은 구현하지 않도록 인터페이스를 쪼갬
 - ex) 전화할 사람에게 전화기능만 있는 폰만 주지 다기능 스마트폰을 줄 필요는 없다

# D(Dependency Inversion Principle)
 - 의존역전 원칙으로 추상화에 의존해야지 구체화에 의존하면 안된다는 뜻
 - 코드는 구체적 클래스가 아니라 추상화 된 인터페이스에 의존해야 나중에 코드를 수정 할 필요가 없다는 뜻
