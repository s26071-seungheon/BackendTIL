# PSA(Portable Service Abstraction)
 - 특정 기술 세부 구현에 얽매이지 않고 스프링이 제공하는 추상화된 인터페이스로 일관된 방식으로 개발 하는 것
 - 외부 라이브러리가 바뀌어도 내 코드는 한 줄도 안고쳐도 되는 것

 ## PSA 역할
 - 추상화 계층의 제공
   - 특정기술(JDBC, JPA등)의 복잡한 로직 노출X 그 위에 단순한 인터페이스 씌움
   - 하위 기술이 무엇이든 상관없이 상위계층에서는 동일한 메서드 호출 방식을 유지하도록 통로 규격화
 - 기술 감추기
   - 상위 코드(비즈니스 로직)가 하위 기술의 세부 사항을 알지 못하게 격리하는 역할
   - 비즈니스 로직이 특정 라이브러리의 클래스나 메서드에 직접 접촉하는 것을 물리적으로 막아버리는 방화벽 수행
 ## PSA의 역할에 의한 장점
 - 기술 독립성 : 특정 기술이나 벤더(Vendor)에 종속되지 않음
   - 벤더 : 제품이나 서비스를 만드는 업체들
     - ex) MySQL, Oracle, AWS, Google Cloud
 - 일관성 : 어떤 서비스를 구현하든 스프링이 정해놓은 일관된 패턴으로 코딩 가능
 - 테스트 용이성 : 실제 기술 대신 테스틍용 서비스로 갈아 끼우기 쉬움

 ## PSA 구성요소
 - 인터페이스(Interface) : 개발자가 사용할 표준화된 입구
 - 구현체(Implementation) : 뒤에서 실제로 돌아가는 구체적인 기술들
 - 설정(Configuration) : 인터페이스에 어떤 구현체를 연결할지 결정
